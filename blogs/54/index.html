Composite Primary Key in Mysql

Oct 11th, 2007

今天有心情弄了一下数据库。虽然之前做的几个Flash网站都有用数据库，不过都没有这么复杂，只是简单的日志和留言的关系。
 
这一次的数据库复杂了许多。今天只是研究了一下怎么在数据库里边表现Friend list。
 
简单来说，本人做的网站的Friend list就是：
整个网站里边有很多个帐号，每个帐号可以加很多个帐号为朋友。所谓的M TO N的关系，多对多。而且还有点奇怪。
跟Michael同学在电话里边讨论了一下，然后再查了一下网上的东西。得出的结果是：用composite primary key实现。
 
一般来说，这种多对多的关系需要3个table来实现。
不过想了半天，因为friend list里边的friend其实也是一个帐号，另外一个table就可以不用了。
 
就两个table: User_acc和UserFriend。
User_acc就是普通的Account table咯，记录一些每个人的信息之类的，用户名，密码什么的。
 
UserFriend实际上可以说是一个Link table…..把User_acc和User_acc联系起来…
很无稽…不过确实是这么一回事儿。要不然就可以理解为 “关系表”，用户与用户的关系。
这个里边主要就两个Column： user_acc_id和friend_acc_id。
这两个column一起作为Primary key…
 
开始还怀疑有可能不行…因为以为
user_acc_id        friend_acc_id                可能会跟                   user_acc_id           friend_acc_id 
1                      2                                                             2                          1
 
一样。
其实不然，composite primary key看来还是很适合这种表现方式。
 
测试了一下得到这个结果：
mysql> create table user_acc
    -> (
    ->  user_acc_id int not null auto_increment,
    ->  username char(40) not null,
    ->  password char(12) not null,
    ->  primary key(user_acc_id)
    -> );
Query OK, 0 rows affected (0.05 sec)
—–
mysql> create table userfriend
    -> (
    ->  user_acc_id int not null,
    ->  friend_acc_id int not null,
    ->  primary key(user_acc_id, friend_acc_id)
    -> );
Query OK, 0 rows affected (0.06 sec)
